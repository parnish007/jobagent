01

It started with rejection emails
I couldn't explain

First year of CS. Everyone around me is applying for internships. I start doing the same. Find a job posting, copy-paste your resume, tweak the summary to mention the company name, hit apply, wait. Get a rejection. Don't know why. Try again.

After a few weeks of this, I had three problems that were genuinely bothering me:

Problem 01

I had no idea what was a good fit. Every job posting looks the same when you're reading ten a day. I couldn't tell which roles were worth applying to versus which were a waste of time.

Problem 02

My resume said the same thing to everyone. I knew you were supposed to tailor it to each job. I also knew that doing it properly would take an hour each time. So I didn't.

Problem 03

I had no idea where I'd applied. A spreadsheet. Then the spreadsheet got out of date. Then I genuinely didn't know if I'd already applied somewhere or not.

I tried LinkedIn Easy Apply. Applied to 30 things in one afternoon. Heard back from zero of them. Faster rejection, same result.

What if I could automate the smart parts too — not just the clicking?

— The idea that started everything
02

The design principle:
AI does the boring work

I wanted a system that would go find relevant jobs every day, read each description and tell me whether I was a good fit, rewrite my resume for each role — not copy-paste, actually rewrite it — and let me review everything before doing anything real.

That last point was important. I'd heard about people building bots that auto-apply to hundreds of jobs. That felt wrong. A real person made the decision to post that job. The least I could do was make a real decision to apply.

03

What I knew going in: not much

I was a first-year student. I'd done the intro Python course. I'd built a few small scripts. I knew what an API was in theory. I had never built a full-stack application. Never touched a database. Didn't know what async meant.

I picked this project specifically because it would force me to learn things I didn't know yet. Simple enough — I was very wrong about the "simple" part.

04

Building it: the real order things happened

1
Getting jobs onto a screen JobSpy + deduplication

Found a Python library called JobSpy that pulled listings from LinkedIn, Indeed, Glassdoor, and ZipRecruiter with a single function call. Then learned about hashing — title + company + url → SHA-256 → unique key. First real problem, first real solution.

2
Making the AI understand me Rubric-based scoring

Built a user profile system with fields for skills, target roles, experience, and preferences. Discovered get_llm_json() — send the prompt, parse the response, retry if JSON comes back malformed. Built a scoring rubric: 40pts skills, 30pts seniority, 20pts domain, 10pts location. Suddenly a score of 72 meant something specific.

3
The async wall Concurrent LLM calls

Spent a week understanding the Python event loop. The mental model that finally clicked: async isn't about doing multiple things simultaneously, it's about not waiting while one thing finishes before starting the next. Rewrote the entire backend to be async. Learned run_in_executor for blocking sync libraries.

4
The human gates The important architectural decision

Used LangGraph's interrupt_before to literally pause execution at named nodes. State is saved to memory. When I come back and say "continue," it picks up exactly where it left off. Two mandatory stops: after scoring (I decide which to apply to), after resume generation (I read before it's submitted).

5
Database design PostgreSQL + pgvector

Key insight: separate raw scraped data from scored data. raw_jobs is never modified. scored_jobs links to it. If I improve the scoring model later, I can re-score existing raw jobs without touching the original data. Added pgvector for cheap cosine similarity instead of expensive LLM calls.

6
Real-time feedback WebSockets + Redis pub/sub

Celery worker publishes events to Redis channel agent:events:{'{user_id}'}. FastAPI WebSocket subscribes and forwards to the browser. React component updates the UI. Falls back to polling every 5 seconds if WebSocket drops. Added a small WiFi icon to show which mode is active.

7
Resume generation + RL DPO fine-tuning

The "do not fabricate" instruction was important to me. Every time I edited the AI's draft, I was giving feedback. Built a preference collection system — edited version is "chosen," original AI draft is "rejected." After 50 comparisons, trigger DPO fine-tuning on flan-t5-base. The infrastructure is there.

# JobSpy: scraping five boards with one call
from jobspy import scrape_jobs
import hashlib

df = scrape_jobs(
    site_name=["linkedin", "indeed", "glassdoor", "zip_recruiter", "google"],
    search_term="ml engineer intern",
    location="Remote"
)

# Deduplication via SHA-256 hash
def make_fingerprint(job):
    key = f"{job['title']}|{job['company']}|{job['job_url']}"
    return hashlib.sha256(key.encode()).hexdigest()

df['fingerprint'] = df.apply(make_fingerprint, axis=1)
df = df.drop_duplicates(subset='fingerprint')

05

What I'd change.
What I'm proud of.

Proud of
The two-gate architecture. It's a real design decision — architecturally impossible to submit without approval.
The LLM abstraction. Switching providers is a settings change, not a code change.
The real-time pipeline: Celery → Redis → FastAPI → browser. Graceful fallback to polling.
Unified setup. One .env.example, one requirements.txt, one virtualenv.
Would change
Use AsyncPostgresSaver instead of MemorySaver for LangGraph. State survives restarts.
Write integration tests from day one. pytest-asyncio against a test database.
Think harder about prompt structure earlier. Read about rubric scoring before writing version one.
